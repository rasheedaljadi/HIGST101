# وثيقة التصميم: جسر تنفيذ الطلبات (Order Fulfillment Bridge)

> **نوع الوثيقة:** تصميم عالي المستوى (High-Level) + تصميم منخفض المستوى (Low-Level) مدمجان.
> **المشروع:** Bagisto 2.4.x (Laravel) — متجر دروبشيبينغ. أول مزوّد تنفيذ هو AliExpress، لكن الطبقة **مستقلة عن المزوّد (provider-agnostic)**.
> **لغة الكود في هذه الوثيقة:** مستقلة عن إطار العمل (Framework-Agnostic)، مع إمكانية تمثيلها بلغة PHP/SQL لملائمة بيئة العمل.

---

## 0. ملاحظة منهجية (اقرأها أولاً)

هذه الوثيقة **مرجع معماري** للمشروع، وليست تلخيصًا لنقاش سابق. كل قرار فيها مبنيّ على قواعد التصميم الهندسي للمنصة واستخلاص البنية الحالية للأنظمة المساعدة:

- خدمات التكامل القائمة (عميل الاتصال وإدارة التوكن والمصادقة).
- دورة حياة طلبات العملاء والفواتير والدفع الإلكتروني والدفع عند الاستلام.
- معايير البنية المستقلة عن الموردين (Provider-Agnostic) وإدارة الكيانات.

**قاعدة صارمة:** حيثما يعتمد قرار على حقيقة لا يمكن التحقق منها من الكود أو من التوثيق الرسمي للمورد الخارجي، سُجِّلت صراحةً كـ **OPEN QUESTION / ASSUMPTION** في القسم 9، ولم يتم اختراع إجابة لها. تحديدًا: تم التأكد عبر بحث عام أن AliExpress **لا توفّر OpenAPI spec عامة**، وأن توثيق Dropshipping Order API محجوب خلف بوابة Open Platform التي تتطلب تسجيل دخول — لذلك أسئلة الـ Idempotency وتعدد الموردين والـ Webhooks وحدود المعدّل تبقى **مفتوحة** ويجب حسمها من التوثيق الرسمي **قبل أي كتابة كود**.

## 1. Overview

**(المقدمة والنطاق)**

جسر تنفيذ الطلبات هو طبقة تربط **طلب العميل الداخلي** (Customer Order) بـ **أمر شراء خارجي لدى المورّد** (Purchase Order / Supplier Order). عند تأكيد الدفع لطلب عميل، يقوم الجسر بإنشاء أمر (أو أكثر) لدى المورّد الخارجي (AliExpress حاليًا)، ثم يتابع حالته ويعكسها على الطلب الداخلي.

المبدأ المعماري الحاكم: **طلب العميل شيء، وأمر الشراء لدى المورّد شيء آخر**. الجسر يترجم بينهما دون دمجهما، وبطريقة **لا تربط وحدة الطلبات بأي مورّد محدد**.

### داخل النطاق
- تحويل طلب عميل مدفوع إلى أمر/أوامر شراء لدى المورّد.
- تعيين الحالات (State Mapping) بين الطرفين.
- منع التكرار داخليًا (internal idempotency) وإعادة المحاولة ومعالجة الفشل.
- تجريد مزوّد التنفيذ عبر واجهة برمجية موحدة (Interface).
- تسجيل وتدقيق العمليات يدوياً وآلياً (Audit Trail vs Attempts).
- توفير لوحة معلومات تشغيلية (Dashboard KPIs) وتصميم متجاوب للواجهات.

### خارج النطاق (حاليًا)
- استرجاع الأموال/المرتجعات الآلية لدى المورّد (RMA).
- تسعير الشحن وقت الدفع (مغطى في تكامل الاستيراد القائم).
- أي منطق دفع؛ الجسر **يستهلك** أحداث الدفع فقط ولا ينشئها.

## 2. ADR-001 — قرار معمارية التنفيذ (Fulfillment Architecture)

**الحالة:** مقبول · **السياق:** Bagisto 2.4.x، دروبشيبينغ متعدد الموردين محتمل.

### القرار
1. **عدم دمج طلب العميل (Customer Order) مع أمر الشراء (Purchase Order).** يُنشأ كيان جديد مستقل.
2. اعتماد **طبقة جسر (Bridge Layer)** بين وحدة المبيعات ووحدة المزوّد.
3. العلاقة **1:N**: طلب عميل واحد ⇐ عدة أوامر شراء (واحد لكل مورّد/متجر/شحنة منطقية).
4. تحديد **مصدر الحقيقة الوحيد (Single Source of Truth)** لكل مرحلة صراحةً.
5. **لا يبدأ إنشاء أمر الشراء إلا بعد الدفع** (مع استثناء الدفع عند الاستلام COD الموثّق في ADR-003).

### لماذا عدم دمج طلب العميل مع أمر الشراء؟
- طلب العميل يملك دورة حياة وحقول مالية وفواتير ومرتجعات وحالات مبيعات خاصة بنظام التجارة الأساسي. دمج بيانات المورّد فيه يعني تعديل الحزم الأساسية وتخريب مسار الترقيات والتحديثات.
- طلب العميل بعملة المتجر وبأسعار البيع؛ أمر الشراء بعملة المورّد وبأسعار التكلفة. خلطهما يُفسد الحسابات المالية وتقارير الأرباح.
- العلاقة ليست 1:1، فلا يمكن تمثيلها بأعمدة إضافية على صف الطلب.

### لماذا طبقة جسر؟
- تعزل «لغة» المتجر (منتجات، SKUs، عناوين) عن «لغة» المورّد (product_id, sku_id, out order id).
- تسمح باستبدال/إضافة موردين دون لمس وحدة المبيعات (ADR-002).
- تجمع منطق منع التكرار وإعادة المحاولة والتعيين في مكان واحد قابل للاختبار.

### لماذا العلاقة 1:N؟
- سلة العميل قد تضم منتجات من **موردين مختلفين** أو **متاجر خارجية مختلفة**؛ كل مورّد يتطلب أمر شراء منفصل.
- لذلك: صف طلب العميل الواحد يرتبط بعدة أوامر شراء، وكل أمر شراء يرتبط بمجموعة بنود تُشتق من بنود طلب العميل الأصلية.

### مصدر الحقيقة (Single Source of Truth) لكل مرحلة

| المرحلة | مصدر الحقيقة | السبب |
|---|---|---|
| السلة والتسعير والدفع | طلب العميل + الفاتورة (Invoices) | نظام التجارة يملك المسار المالي بالكامل. |
| قرار «هل نبدأ التنفيذ؟» | الجسر (بناءً على حدث تأكيد الفاتورة) | منطق الأعمال يخص الجسر لا المبيعات. |
| حالة أمر الشراء وبياناته لدى المورّد | سجل أمر الشراء المحلي (نسخة معكوسة من المورّد) | المورّد هو الحقيقة النهائية، لكن نُخزّن نسخة محلية موثوقة نُزامنها. |
| حالة التتبع/الشحن لدى المورّد | المورّد الخارجي، ويُعكَس إلى النظام عبر الاستعلام (Polling) | لا نملك بيانات الشحن الفعلية. |
| الحالة الظاهرة للعميل | طلب العميل (تُحدَّث اشتقاقًا من حالات أوامر الشراء) | العميل يرى طلبه فقط. |

## 3. ADR-002 — تجريد مزوّد التنفيذ (Fulfillment Provider Abstraction)

**الحالة:** مقبول.

### القرار
تعريف واجهة برمجية موحدة للمزودين (FulfillmentProviderInterface) تعتمد عليها وحدة الجسر فقط. **لا تعرف وحدة الطلبات ولا الجسر عن المورد الخارجي شيئًا**؛ يعرفانه فقط عبر هذا العقد وعبر مُعرّف نصّي (Provider Code).

### العقد البرمجي الموحد (Conceptual Interface)

```
Interface FulfillmentProviderInterface
{
    Function code() -> String
    Function isConfigured() -> Boolean
    Function createSupplierOrder(request: SupplierOrderRequest) -> SupplierOrderResult
    Function getSupplierOrderStatus(externalOrderId: String) -> SupplierOrderStatus
    Function findByReference(internalReference: String) -> String or Null
    Function cancelSupplierOrder(externalOrderId: String) -> SupplierOrderResult
}
```

## 4. ADR-003 — سياسة تحفيز تنفيذ الطلب (Order Fulfillment Trigger Policy)

**الحالة:** مقبول.

### القرار
الاستماع إلى حدث حفظ الفاتورة (Invoice Save Event) كحدث إطلاق أساسي للتنفيذ.

- إتمام إنشاء الفاتورة يعني **استلام/تأكيد الدفع** فعليًا للمبلغ.
- بالنسبة لـ COD: إذا كان توليد الفواتير التلقائي مفعّلًا في إعدادات طريقة الدفع، يمر COD أيضًا عبر نفس الحدث لتوحيد المسار. وفي حال إيقاف التوليد التلقائي لـ COD، ينتظر النظام إنشاء الفاتورة يدوياً لإطلاق التنفيذ.

## 5. Architecture & Flow

### 5.1 مخطط دورة الحياة الكامل (Lifecycle)

```
[دفع العميل] ──> [إنشاء الفاتورة] ──> [حدث حفظ الفاتورة]
                                            │
                                            ▼
                                  [تجميع البنود حسب المورد]
                                            │
                                            ▼
                                [إنشاء مفتاح منع التكرار]
                                            │
                                            ▼
                                 [طابور المهام الخلفية]
                                            │
                                            ▼
                                [التأكد من خلو القفل المفتوح]
                                            │
                                            ▼
                             [الاستعلام بقيمة المرجع الداخلي]
                                            │
                             ┌──────────────┴──────────────┐
                             ▼ (موجود مسبقاً)               ▼ (غير موجود)
                      [ربط أمر الشراء]             [إرسال طلب الإنشاء للمورد]
                                                           │
                                             ┌─────────────┴─────────────┐
                                             ▼ (نجاح)                    ▼ (فشل)
                                      [حفظ البيانات والتتبع]       [تصنيف الخطأ وإعادة المحاولة]
```

---

### 5.2 تعيين الحالات وسلوك الواجهة (UI State Machine Behavior)

تم تعيين حالات المورّد وحالات الجسر، مع تحديد السلوك البصري والعمليات المسموحة في واجهة الإدارة:

| حالة الجسر (PO State) | المعنى التشغيلي | تعيينها على طلب العميل | الأزرار المفعّلة في الواجهة (UI Actions) | رسالة الحالة الظاهرة للأدمن | لون الشارة (Badge) |
|---|---|---|---|---|---|
| `pending` | بالانتظار للتنفيذ الآلي | قيد المعالجة | لا يوجد (معالجة خلفية) | قيد الانتظار للتنفيذ التلقائي... | برتقالي |
| `submitting` | جارٍ التخاطب مع المورد | قيد المعالجة | لا يوجد (مغلق للحماية) | جاري إرسال الطلب للمورد ومنع التكرار... | أزرق فاتح |
| `submitted` | تم الإنشاء، بانتظار الشحن | قيد المعالجة | تحديث الحالة، إلغاء أمر الشراء | تم إنشاء أمر الشراء بنجاح وبانتظار شحن المورد. | أزرق داكن |
| `awaiting_payment_to_supplier` | بانتظار سداد البضاعة للمورد | قيد المعالجة | تحديث الحالة، إلغاء أمر الشراء | مقبول لدى المورد؛ يرجى إتمام الدفع للمورد. | بنفسجي |
| `shipped` | تم الشحن من المورد | قيد المعالجة | تحديث الحالة | تم شحن الطلب. رقم التتبع متوفر. | نيلي |
| `delivered` | وصل للعميل النهائي | مكتمل | لا يوجد (حالة نهائية) | تم تسليم الطلب بنجاح للعميل. | أخضر |
| `needs_manual_review` | فشل دائم أو استنفاد محاولات | يبقى قيد المعالجة | إعادة محاولة يدوية، إلغاء الطلب، تعديل البيانات | فشل التنفيذ الآلي ويتطلب مراجعة بشرية. | أحمر |
| `canceled` | أُلغي لدى المورد أو محليًا | ملغى/يتبع السياسة | لا يوجد (حالة نهائية) | تم إلغاء تنفيذ هذا الأمر. | رمادي |

### 5.3 معالجة الفشل وتصنيف الأخطاء (Error & Failure Handling)

لإتاحة التقارير والفرز الدقيق للأخطاء البرمجية والتشغيلية، يتم تصنيف الأخطاء باستخدام نوع بيانات تعداد فريد ومحدّد (Strongly Typed Enumeration) يسمى `FulfillmentErrorType`:

* **قيم التعداد (Enum Values)**:
  * `PROVIDER_ERROR`: أخطاء تأتي من تطبيقات وواجهات المورد نفسه (مثل حساب مجمد أو خطأ خادم داخلي لدى المورد).
  * `NETWORK_ERROR`: أخطاء الشبكة والاتصال والـ timeouts.
  * `AUTH_ERROR`: فشل المصادقة أو انتهاء توكنات الوصول وفشل تجديدها.
  * `VALIDATION_ERROR`: أخطاء التحقق من البيانات (مثل عنوان خاطئ أو حقل مطلوب مفقود).
  * `BUSINESS_RULE_ERROR`: أخطاء تتعلق بقواعد العمل والتجارة (مثل فروق أسعار كبيرة تتجاوز المسموح به، أو نفاد مخزون المنتج).
  * `MANUAL_REVIEW`: الأخطاء التي تفرض المراجعة اليدوية أو التي يرفعها الموظف يدوياً.

يتم تخزين نوع الخطأ في حقل مخصص في سجل المحاولات `error_type` لتسهيل التحليل والفرز لاحقاً.

### 5.4 تصنيف الإجراءات التشغيلية (Operations Actions Classification)

يحدد هذا الجدول تصنيف كافة العمليات التي تجري في لوحة الإدارة وتأثيرها المعماري:

| الإجراء | طريقة التنفيذ (Execution) | يحتاج تأكيد؟ (Confirmation) | قابل للتراجع؟ (Reversible) | يسجل في سجل التدقيق؟ (Audit Log) | الصلاحية المطلوبة (RBAC Permission) | مسار الموافقة (Approval Workflow) |
|---|---|---|---|---|---|---|
| **إعادة محاولة يدوية (Manual Retry)** | غير متزامن (خلفي عبر طابور المهام) | لا | لا | نعم | صلاحية إعادة المحاولة | تنفيذ فوري |
| **إلغاء أمر الشراء (Force Cancel)** | غير متزامن (استدعاء المورد في الخلفية) | نعم (تنبيه تحذيري) | لا | نعم | صلاحية الإلغاء | **اختياري/مؤسسات:** يتطلب موافقة إذا تم الدفع للمورد |
| **تحديث الحالة يدوياً (Sync Status)** | غير متزامن (استعلام فوري عن حالة المورد) | لا | لا ينطبق | نعم | صلاحية التحديث والتزامن | تنفيذ فوري |
| **تعديل البيانات يدوياً (Manual Edit)** | متزامن (تعديل البيانات محلياً قبل التقديم) | نعم (تأكيد الحفظ) | نعم | نعم | صلاحية التعديل | **اختياري/مؤسسات:** يتطلب موافقة إذا تم تقديم الأمر مسبقاً |
| **تخطي الحالة يدويًا (State Override)** | متزامن (تغيير الحالة يدوياً في قاعدتنا) | نعم (تأكيد مشدد) | لا | نعم | صلاحية التخطي والموافقة | **إلزامي الموافقة دائماً** (مدير النظام) |

## 6. Low-Level Design & Data Models

### 6.1 نماذج البيانات المعدلة والجديدة (Conceptual Database Schemas)

#### 1. جدول أوامر الشراء `purchase_orders`
* `id`: مفتاح رئيسي فريد.
* `order_id`: معرّف الطلب في نظام التجارة الأساسي.
* `provider`: كود الموفر النصي (مثل "aliexpress").
* `supplier_signature`: معرّف متجر المورد.
* `idempotency_key`: مفتاح منع التكرار الفريد (طبقة المنع 1).
* `internal_reference`: المرجع الفريد المرسل للمورد.
* `external_order_id`: رقم الطلب الخارجي.
* `state`: الحالة التشغيلية للجسر.
* `supplier_state_raw`: حالة المورد الخام.
* `attempts`: عدد محاولات الاتصال.
* `last_error`: تفاصيل آخر خطأ.
* `tracking_number`: رقم التتبع.
* `tracking_company`: شركة الشحن.
* `supplier_cost`: تكلفة الشراء من المورد.
* `supplier_currency`: عملة الشراء من المورد.
* `payload_snapshot`: اللقطة الهيكلية للبيانات.
* `submitted_at`: تاريخ التقديم للمورد.
* `timestamps`: تاريخ الإنشاء والتعديل.

#### 2. جدول محاولات التخاطب `fulfillment_attempts` (مخصص للمحاولات الآلية للـ API)
* `id`: مفتاح رئيسي.
* `purchase_order_id`: معرّف أمر الشراء.
* `attempt_no`: رقم المحاولة الحالية.
* `result`: النتيجة (نجاح، خطأ مؤقت، خطأ دائم).
* `error_type`: نوع الخطأ (Strongly Typed Enumeration).
* `provider_code`: كود الخطأ الخام من المورد.
* `message`: نص الخطأ (خالٍ من الأسرار ومقصوص).

#### 3. سجل تدقيق عمليات الموظفين `fulfillment_audit_logs` (مخصص لتصرفات البشر داخل لوحة الإدارة)
* `id`: مفتاح رئيسي.
* `purchase_order_id`: معرّف أمر الشراء المرتبط (nullable).
* `user_id`: معرّف المشرف أو الموظف المنفّذ للإجراء.
* `action`: الإجراء المتخذ (إعادة محاولة، إلغاء، تخطي حالة، تعديل).
* `reason`: سبب الإجراء المكتوب يدوياً (مطلوب للعمليات الحساسة).
* `ip_address`: عنوان IP للمنفذ.
* `changes_payload`: الجوانب التي تغيرت (قبل وبعد الإجراء).
* `approval_status`: حالة الموافقة (pending_approval, approved, rejected, executed).
* `approved_by`: معرّف المدير أو المسؤول الذي وافق على الإجراء (nullable).

### 6.7 معمارية جدول البيانات (DataGrid Architecture)

لوحة تحكم تنفيذ الطلبات تعتمد على تصميم DataGrid مجرد بالكامل لتقليل الاعتماد على مورد واحد:

#### تقسيم الأعمدة:
1. **الأعمدة الافتراضية (Default Columns)**: معرّف أمر الشراء، رقم طلب العميل، المزود (Provider Name)، متجر المورد، التكلفة، الحالة، رقم التتبع، وتاريخ الإرسال.
2. **الأعمدة الاختيارية (Optional Columns)**: المرجع الداخلي، رقم الطلب الخارجي، عدد المحاولات، وآخر خطأ مخزن.
3. **الأعمدة المخفية (Hidden Columns)**: مفتاح منع التكرار، واللقطة الهيكلية للبيانات.

#### طرق الفلترة وعرض القوائم (Saved Views):
* **لوحة موظف التنفيذ (Operator View)**:
  * **طابوري (My Queue)**: يعرض الأوامر المخصصة للموظف أو التي بانتظار الإرسال.
  * **يتطلب إجراء (Needs Review)**: يعرض الأوامر المتعثرة في حالة مراجعة يدوية.
  * **فشل اليوم (Failed Today)**: الأوامر الفاشلة خلال الـ 24 ساعة الأخيرة.
* **لوحة المدير (Manager / Executive View)**:
  * **تحليل الموردين (Provider Health & Analytics)**: إحصائيات النجاح والفشل لكل مورد.
  * **طلبات تجاوزت المدة المقررة (SLA Breaches)**: الأوامر التي لم تشحن بعد مرور 48 ساعة من تقديمها.

### 6.8 تجريد الواجهة ودعم تعدد الموردين (Multi-Provider Readiness)

- **صارم:** يمنع عرض أي واجهات تحتوي على مسميات مرتبطة بمورد واحد مثل "AliExpress ID" في الشاشات العامة.
- **تطبيق التجريد:** تستخدم الواجهة تعبيرات عامة مثل `Provider` و `External Order ID`. عند تحميل البيانات، يتم الاستعلام من الواجهة الموحدة للمزود لعرض المورد ديناميكياً (مثال: "Provider: AliExpress" أو "Provider: CJ Dropshipping").

---

### 6.9 التصميم المتجاوب للواجهات (Responsive Behavior)

لتسهيل إدارة العمليات من الأجهزة اللوحية والهواتف الذكية، يتم تطبيق القواعد التالية:

1. **أولويات إخفاء الأعمدة (Column Prioritization)**:
   - **الأولوية 1 (شاشات الهواتف)**: رقم أمر الشراء، رقم طلب العميل، الحالة، رقم التتبع، تكلفة الطلب.
   - **الأولوية 2 (أجهزة التابلت)**: يضاف إليها اسم المزود (Provider)، واسم المتجر.
   - **الأولوية 3 (شاشات سطح المكتب فقط)**: يضاف المرجع الداخلي، وتاريخ الإرسال، وعدد المحاولات.
2. **التمدد الرأسي (Row Accordion)**: في شاشات الجوال، يؤدي النقر على أي صف في الـ DataGrid إلى تمدده لأسفل لعرض الأعمدة المخفية (الأولوية 2 و 3) كقائمة بيانات رأسية مفصلة.
3. **قائمة الإجراءات المختصرة (Actions Dropdown)**: تُجمع أزرار العمليات (إلغاء، إعادة محاولة، مزامنة) في زر منسدل واحد (More Actions) لمنع التداخل والازدحام الأفقي في الشاشات الصغيرة.

## 7. معمارية الإشعارات والتنبيهات (Notification Severity)

يتم تصنيف الإشعارات التي يطلقها الجسر بناءً على درجة الخطورة لضمان كفاءة العمليات دون إزعاج الموظفين:

| المستوى (Severity) | الحدث المسبب | القناة المستهدفة والإجراء المتبع |
|---|---|---|
| **معلومات (Info)** | - بدء إعادة محاولة يدوية.<br/>- حفظ تعديل يدوي للعنوان. | **Toast Notification** تظهر وتختفي تلقائياً للأدمن المنفذ فقط. |
| **تحذير (Warning)** | - تأخر Polling لمزامنة الحالات لأكثر من 30 دقيقة.<br/>- بقاء أمر الشراء في حالة إرسال معلقة لأكثر من 15 دقيقة. | **Badge Counter** يزداد في القائمة الجانبية + إشعار صامت في مركز الإشعارات. |
| **خطأ (Error)** | - فشل دائم لأمر شراء (مثال: عنوان غير صالح).<br/>- استنفاد كافة المحاولات الآلية للطلب ودخوله مراجعة يدوية. | **إرسال بريد إلكتروني** للمشرفين + **Admin Notification Banner** ثابت في لوحة التحكم للطلب المعني. |
| **حرج (Critical)** | - فشل تجديد توكن المصادقة التلقائي وتوقف الاتصال بالمورد.<br/>- حظر الـ API بالكامل من المورد بسبب تجاوز حد المعدل.<br/>- توقف المورد بالكامل عن الاستجابة. | **شريط أحمر ثابت (Critical Alert Bar)** في أعلى لوحة تحكم الإدارة لجميع المدراء + بريد إلكتروني فوري عاجل. |

## 8. مؤشرات الأداء للوحة التحكم (Dashboard KPIs)

تحتوي لوحة تحكم وحدة التنفيذ على المؤشرات التالية مع تحديد مصدر بياناتها وطريقة حسابها بدقة:

### 1. نسبة نجاح التنفيذ (Success Rate)
- **الحساب**: `Count(state IN (submitted, shipped, delivered)) / Total POs`
- **المصدر**: جدول أوامر الشراء لفلترة التواريخ المحددة.

### 2. نسبة إعادة المحاولة (Retry Rate)
- **الحساب**: `Count(attempts > 1) / Total POs`
- **المصدر**: جدول أوامر الشراء أو جدول محاولات التخاطب مع المورد.

### 3. متوسط وقت التنفيذ (Average Fulfillment Time)
- **الحساب**: `AVG(submitted_at - invoices.created_at)`
- **المصدر**: ربط جدول أوامر الشراء مع جدول الفواتير للطلبات الناجحة.

### 4. الطلبات المعلقة (Orders Waiting)
- **الحساب**: `Count(state IN (pending, submitting))`
- **المصدر**: استعلام فوري من جدول أوامر الشراء.

### 5. طلبات قيد المراجعة اليدوية (Orders in Manual Review)
- **الحساب**: `Count(state = needs_manual_review)`
- **المصدر**: استعلام فوري من جدول أوامر الشراء.

### 6. صحة الموردين (Provider Health)
- **الحساب**: يعتمد على صلاحية توكن المصادقة الحالي ومعدل نجاح آخر 50 محاولة اتصال بالـ API (إذا قل معدل النجاح عن 90% تعتبر الحالة Degraded، وإذا انقطع الاتصال بالكامل تعتبر Offline).
- **المصدر**: سجلات المصادقة وجدول محاولات التخاطب.

### 7. تراكم الطابور (Queue Backlog)
- **الحساب**: استعلام مباشر لعدد المهام البرمجية المعلقة في طابور المعالجة.
- **المصدر**: قاعدة بيانات المهام الخلفية.

### 8. تأخر الاستعلام الدوري (Polling Delay)
- **الحساب**: الفرق الزمني بين تاريخ التشغيل المخطط الفعلي لمهمة الاستعلام والوقت الحالي.

## 11. ميزانية الأداء ومتطلبات الاستعلامات (Performance Budget)

لضمان عمل المتجر بسلاسة واستقرار خوادم قواعد البيانات، تلتزم وحدة التنفيذ بالأحكام التالية:

1. **تحميل لوحة البيانات (Dashboard Load Time)**: يجب ألا يتجاوز زمن تحميل لوحة البيانات **ثانيتين (2s)**.
   * **السياسة**: يتم حساب مؤشرات الأداء بشكل دوري خلفي (كل 15 دقيقة) ويتم تخزين النتائج في الكاش (Cache). الواجهة تقرأ دائماً من الكاش لتفادي الاستعلامات الثقيلة.
2. **منع استعلامات N+1 (N+1 Query Prevention)**:
   * **صارم**: يمنع تماماً الاستدعاء المتكرر للعلاقات داخل الحلقات التكرارية للبيانات. يجب دائماً استخدام تقنيات التحميل المسبق للعلاقات (Eager Loading) المتاحة في إطار العمل لجلب البيانات في استعلامات مجمعة ومحسنة عند استرجاع القوائم والـ DataGrids.
3. **دعم حجم البيانات الكبير (100k+ Records)**:
   * يجب أن يدعم جدول البيانات فلترة وتصفح أكثر من 100,000 سجل دون بطء.
   * **الحل**: فرض Pagination إلزامي دائم في الاستعلامات، والتأكد من وجود فهارس (Indexes) مناسبة على حقول البحث والترتيب في قاعدة البيانات.

## 12. رايات المزايا والإطلاق التدريجي (Feature Flags Policy)

للسماح بالإطلاق التجريبي التدريجي للمنتج، تُضاف إعدادات التحكم التالية لملف إعدادات المزايا:

- **الرايات المدعومة**:
  * `FULFILLMENT_ADMIN_UI_ENABLED`: يتيح للمشرفين إخفاء أو إظهار شاشات وقوائم وحدة التنفيذ في لوحة الإدارة بالكامل بلمسة واحدة.
  * `FULFILLMENT_RETRY_ENABLED`: يتيح إيقاف طابور إعادة المحاولة الآلية فورًا في حال حدوث عطل عام بالخادم لتجنب تكرار الأخطاء.
  * `FULFILLMENT_MANUAL_CANCEL_ENABLED`: خيار أمان يمنع الموظفين من إلغاء أوامر الشراء يدوياً لتجنب مشاكل استرجاع الأموال مع الموردين، ويحصرها بالمدراء فقط.

- **سياسة الإدارة والتنظيف (Cleanup Policy)**:
  * **الإطلاق المؤقت**: تعتبر رايات المزايا مؤقتة لتسهيل النشر والتجريب التدريجي للوحدة.
  * **استقلالية الأعمال**: يمنع تمامًا ربط منطق الأعمال المالي والتشغيلي الأساسي للطلبات بتعطيل هذه الرايات بشكل دائم.
  * **التنظيف والفرز**: بعد استقرار الميزات في بيئة التشغيل الفعلية (لفترة لا تتجاوز 90 يومًا)، يجب جدولة إزالة هذه الرايات من الكود البرمجي والملفات لتفادي زيادة الكود غير المستخدم (Code Debt).

## 13. مسار موافقة العمليات الحساسة (Approval Workflow - Enterprise Option)

للحد من المخاطر المالية والتشغيلية، تخضع العمليات الحساسة (مثل الإلغاء اليدوي لأمر شراء مدفوع، أو فرض تعديل كميات البنود، أو التخطي الإجباري لحالة الطلب) لمسار موافقة اختياري (قدرة متقدمة للمؤسسات / Optional Enterprise Capability):

- **قابلية الضبط**: يمكن تفعيل أو إلغاء تفعيل هذا المسار بالكامل عبر ملف إعدادات الوحدة ليناسب المتاجر الصغيرة التي لا تحتاج إلى تعقيد في العمليات التشغيلية.
- **في حال التفعيل**:
  1. **تقديم الطلب (Request)**: عند قيام الموظف بالإجراء الحساس، لا يتم تنفيذه فورًا. بدلاً من ذلك، يتم إنشاء سجل في جدول سجل التدقيق وتكون الحالة `approval_status = 'pending_approval'`.
  2. **الإخطار (Notification)**: يتم إخطار المدراء بوجود طلب موافقة تشغيلي معلق يحتاج مراجعة.
  3. **الموافقة والرفض (Approval / Rejection)**:
     - في حال **الرفض**: يتم تغيير الحالة لـ `rejected` ويسجل سبب الرفض، ويتم تنبيه الموظف.
     - في حال **الموافقة**: يتم تغيير الحالة لـ `approved` ثم `executed` ويتم تشغيل المهمة فورًا، مع تسجيل معرّف المشرف الموافِق في سجل التدقيق.

## 14. النسخ والتطور المستقبلي (Versioning & Future Evolution)

لضمان صلاحية الوثائق المعمارية لسنوات قادمة وضمان قدرتها على استيعاب التوسعات المستقبلية، تلتزم هندسة الجسر بالقواعد التالية:

### 1. إضافة موردين جدد (Adding New Providers)
* **استخدام العقد البرمجي**: تتم إضافة أي مورد جديد عن طريق إنشاء محول (Adapter) جديد يطبق الواجهة البرمجية الموحدة `FulfillmentProviderInterface`.
* **العزل**: يمنع تماماً حقن أي أكواد برمجية تخص المورد الجديد في المكونات المبيعية أو المشتركة.
* **التسجيل**: يتم تسجيل المورد الجديد في ملف الإعدادات المخصص، ويتولى سجل الموزدين حقنه تلقائيًا دون تغيير في البنية الأساسية.

### 2. إضافة حالات تشغيلية جديدة (Extending States)
* **ثبات الحالات الأساسية**: يجب الإبقاء على الحالات الرئيسية للجسر في سجل الحالات الموحد ثابتة لمنع كسر توافق الحزم التجارية الأخرى.
* **الحالات الفرعية للموردين**: إذا قدم المورد الجديد حالات خاصة به، يتم تعيينها (Mapping) داخل محول المورد إلى إحدى الحالات التسع الأساسية للجسر في القسم 5.3، أو يتم تسجيلها في عمود `supplier_state_raw` لأغراض التدقيق دون كسر الحالة العامة.

### 3. التوافق الخلفي (Backward Compatibility Rules)
* **ثبات الواجهات (Interface Immutability)**: يمنع تعديل توقيعات الواجهات البرمجية (Method Signatures) في `FulfillmentProviderInterface` بعد اعتمادها؛ وفي حال الضرورة يتم إنشاء إصدار جديد من الواجهة (مثال: `FulfillmentProviderV2Interface`).
* **أمان تعديل قاعدة البيانات (Database Migrations Safety)**: يمنع استخدام عمليات تعديل الجداول الأساسية التي قد تؤدي لقفل الجداول أو مسح البيانات في بيئة الإنتاج. يجب أن تكون كافة إضافات الجداول والأعمدة متوافقة تراجعياً (Nullable columns or with default values).

## 15. دورة حياة التوثيق والخطوات التالية (Document Lifecycle)

- **الوضعية الحالية**: تنتهي هذه المرحلة باعتماد هذا التصميم ومستند المتطلبات كمرجع رسمي للعمليات المعمارية.
- **الخطوة التالية (قبل البرمجة مباشرة)**: سيتم إنشاء **Implementation Plan** تفصيلي منفصل، يحدد بدقة:
  - مسارات الملفات الجديدة والمعدلة.
  - ملفات التهجير (Migrations) الخاصة بالجداول الثلاثة الجديدة والمعدلة.
  - تعريف مسارات الـ Routes، الـ Controllers، الـ DataGrids، والـ Blade Components.
  - خطة تغطية الاختبارات الآلية (Automations & Pest Tests).
  - خطة التراجع (Rollback Plan) في حال حدوث طوارئ أثناء النشر.
